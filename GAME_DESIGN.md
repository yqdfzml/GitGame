# Git 游戏设计说明书

## 1. 游戏愿景

这个项目不是一个单纯的 Git 命令速查工具，而是一个面向初学者的互动式 Git 修炼游戏。玩家通过阅读任务、输入 Git 命令、获得即时反馈、完成关卡、获取经验和解锁称号，逐步建立对 Git 工作流的直觉。

推荐的产品定位是：

> 一个面向 Git 初学者的中文互动闯关游戏，用“代码修仙 / 命令江湖”的轻量主题，把 `add`、`commit`、`branch`、`merge`、`reset`、`conflict` 等 Git 概念变成可练习、可反馈、可成长的关卡体验。

核心设计目标：

- **低压力学习**：输错命令不是失败，而是一次获得解释的机会。
- **即时反馈**：每次输入都应该让玩家更理解当前 Git 状态。
- **渐进成长**：从基础提交到分支、合并、冲突、历史恢复逐步推进。
- **轻度游戏化**：XP、等级、评分和称号提供成就感，但不压过学习本身。
- **中文语境友好**：文案轻松、有鼓励感，同时保持 Git 概念准确。

## 2. 目标用户

### 2.1 Git 初学者

这类玩家可能刚接触版本控制，不熟悉工作区、暂存区、提交、分支等概念。游戏需要帮助他们把抽象概念和具体命令建立联系。

### 2.2 新手开发者

这类玩家已经会写代码，但对团队协作中的 Git 流程不稳定，容易害怕 merge、reset、conflict 等操作。游戏需要提供安全的模拟环境，让他们敢于尝试。

### 2.3 教学场景用户

讲师、助教、训练营或社团可以把游戏作为课堂演示和课后练习工具。关卡需要足够清晰，反馈需要能帮助复盘。

## 3. 当前 MVP 基础

当前项目已经具备一个可扩展的 Git 学习游戏雏形：

- `src/App.tsx`：已有 Home、Levels、Play、Profile 四类页面流程。
- `src/game/challenges.ts`：已有 6 个线性挑战，覆盖首次提交、暂存、分支、合并、历史回溯和冲突。
- `src/game/growth.ts`：已有 XP、等级、评分、重复挑战收益控制等成长逻辑。
- `src/game/titles.ts`：已有中文修仙风格的称号与解锁规则。
- `src/game/storage.ts`：已有本地存档能力。
- `GIT_FLOW.md`：已有 MVP 流程和验收目标。

后续设计应优先尊重这些已有能力，在现有基础上扩展，而不是推翻重做。

## 4. 核心游戏循环

推荐核心循环如下：

1. 玩家从首页或关卡页进入挑战。
2. 阅读剧情背景、任务目标和当前 Git 场景。
3. 在模拟终端中输入 Git 命令。
4. 系统判断命令是否正确、是否符合当前步骤、是否顺序合理。
5. 系统给出成功、错误、顺序偏差或提示反馈。
6. 玩家完成全部目标后进入结算。
7. 系统发放 XP、评分、最佳成绩更新和称号解锁。
8. 玩家返回关卡页或继续下一关。

设计原则：

> 每一次命令输入都应该产生明确反馈；每一次成功都应该让玩家感觉自己更理解 Git，而不仅仅是通过了一个判断条件。

## 5. UI 设计

### 5.1 整体信息架构

当前页面结构可以继续保留，并在设计上明确每个页面的职责：

- **Home**：建立游戏主题和继续学习入口。
- **Levels**：展示学习路径、关卡状态和解锁进度。
- **Play**：承载核心命令输入、任务反馈和通关体验。
- **Profile**：展示玩家成长、称号、成就和学习成果。

### 5.2 首页 Home

首页负责让玩家快速理解“我是谁、我到哪里了、下一步做什么”。

建议内容：

- 游戏标题，例如“Git 修仙录”。
- 副标题，例如“从凡人开发者到版本控制宗师”。
- 当前等级、当前称号、总 XP。
- 等级进度条，例如“Lv.3 · 40/100 XP”。
- 主按钮：“继续修炼”。
- 次按钮：“查看关卡”“我的道号”。
- 最近一次挑战或推荐下一关。

交互重点：

- 如果玩家有未完成进度，主按钮直接进入下一关。
- 如果玩家首次进入，主按钮进入第一关，并展示简短新手说明。
- 首页不应堆太多说明，重点是建立目标感和继续行动。

### 5.3 关卡页 Levels

关卡页负责展示学习路径，让玩家知道当前已掌握什么、下一步要学什么。

关卡卡片建议包含：

- 关卡编号。
- 关卡标题。
- 所属章节。
- Git 技能点。
- 难度。
- 基础 XP。
- 完成状态。
- 最佳评分。
- 解锁状态。

关卡状态建议：

- **已完成**：显示最佳评分和“再修炼”。
- **已解锁未完成**：显示“开始挑战”。
- **未解锁**：显示锁定状态和解锁条件。

交互重点：

- 鼠标悬浮或展开时展示学习目标摘要。
- 未解锁关卡不要只显示“锁定”，要告诉玩家“完成上一关后解锁”。
- 已完成关卡允许重复挑战，但需明确重复挑战的收益规则。

### 5.4 游戏页 Play

游戏页是核心体验，应优先保证信息清晰和输入流畅。

推荐布局：

- 左侧：任务说明区。
- 右侧：模拟终端区。
- 底部或侧栏：目标进度、提示、当前状态。

任务说明区包含：

- 剧情背景。
- 当前任务目标。
- Git 概念解释。
- 已完成目标列表。
- 当前建议思考方向。

模拟终端区包含：

- 命令输入框。
- 历史命令记录。
- 成功反馈。
- 错误反馈。
- 模拟 Git 输出。
- 当前仓库状态摘要。

交互重点：

- 输入框始终是页面焦点核心。
- 正确命令应有明确的推进反馈。
- 错误命令应解释原因，而不是只显示错误。
- 允许玩家查看提示，但提示会影响最终评分。
- 完成关卡后弹出结算面板，不要突然跳转。

### 5.5 个人页 Profile

个人页负责强化长期成长感，让玩家看到“我学会了什么”。

建议内容：

- 当前等级。
- 当前称号。
- 总 XP。
- 已完成关卡数量。
- 已解锁称号列表。
- 每个 Git 能力方向的掌握情况。
- 最近完成挑战。
- 重置存档入口。

能力标签示例：

- 提交基础。
- 暂存区理解。
- 分支操作。
- 合并流程。
- 历史回溯。
- 冲突处理。
- 远程协作。
- 恢复与撤销。

## 6. 用户交互设计

### 6.1 命令输入

玩家通过模拟终端输入 Git 命令。当前 MVP 已经支持命令序列判断，后续可以逐步支持更宽容的命令匹配。

当前阶段建议继续保持清晰的标准答案路径，例如：

```bash
git add .
git commit -m "first commit"
git branch feature/login
git merge feature/login
```

未来可扩展为目标导向判断：

- `git add .` 与 `git add README.md` 在特定任务中都可接受。
- `git checkout -b feature/x` 与 `git switch -c feature/x` 可以视作等价路径。
- commit message 可以允许不同文本，只要结构合理。

### 6.2 正确反馈

正确反馈应同时表达“做对了”和“为什么这一步有效”。

示例：

> 成功。文件已经进入暂存区，下一次 commit 会把这些改动写入版本历史。

### 6.3 错误反馈

错误反馈不应只说“命令错误”。它应该解释：

- 当前缺少哪一步。
- Git 为什么不能执行玩家想做的事。
- 玩家下一步应该思考什么。

示例：

> 还差一步。当前文件还没有进入暂存区，`git commit` 暂时找不到可以提交的内容。先试试把改动加入暂存区。

### 6.4 顺序反馈

当前 MVP 已经区分“命令正确但顺序不完全正确”的情况。这个设计值得保留，因为它能鼓励玩家理解 Git 工作流，而不是只背命令。

反馈示例：

> 这个命令本身是有效的，但现在还不是最合适的时机。Git 操作通常需要先确认状态，再执行下一步。

### 6.5 分层提示

提示建议设计为三层：

1. **概念提示**：解释当前 Git 概念。
2. **方向提示**：告诉玩家需要进行哪类操作。
3. **接近答案提示**：给出命令结构或关键命令。

示例：

- 提示 1：提交前需要先告诉 Git 哪些文件要进入快照。
- 提示 2：你需要使用 `git add`。
- 提示 3：可以尝试 `git add .`。

提示设计原则：

- 提示要帮助理解，不只是泄露答案。
- 每次使用提示会轻微影响评分，但不应阻止通关。
- 新手关可以更慷慨，高阶关可以更克制。

### 6.6 结算反馈

关卡完成后展示结算面板。

建议内容：

- 通关成功文案。
- 本次评分。
- 获得 XP。
- 是否刷新最佳成绩。
- 是否升级。
- 是否解锁新称号。
- 本关学到的 Git 概念。
- 下一关入口。

结算的重点不是炫耀数值，而是让玩家明确“我刚刚掌握了什么”。

## 7. 关卡设计

### 7.1 当前 MVP 关卡

当前 6 个关卡可以整理为第一阶段学习路径。

#### 关卡 1：初次提交

- 学习目标：理解仓库初始化、暂存和提交。
- 核心命令：`git init`、`git add`、`git commit`。
- 教学重点：commit 是一次版本快照，提交前需要先暂存。
- 通关反馈：玩家完成第一次版本记录，正式入门 Git。

#### 关卡 2：暂存区修炼

- 学习目标：理解工作区、暂存区、提交区的区别。
- 核心命令：`git status`、`git add`、`git commit`。
- 教学重点：Git 不会自动提交所有文件，暂存区决定下一次提交包含什么。
- 通关反馈：玩家掌握“先选择，再提交”的基本节奏。

#### 关卡 3：分支初探

- 学习目标：理解分支是独立开发线。
- 核心命令：`git branch`、`git checkout` 或 `git switch`。
- 教学重点：分支让玩家可以在不影响主线的情况下开发功能。
- 通关反馈：玩家创建并切换到新分支。

#### 关卡 4：合并之术

- 学习目标：理解把功能分支合并回主线。
- 核心命令：`git checkout main` / `git switch main`、`git merge`。
- 教学重点：合并是把另一条开发线的成果带回当前分支。
- 通关反馈：玩家完成一次基础分支协作流程。

#### 关卡 5：历史回溯

- 学习目标：理解提交历史和回退操作。
- 核心命令：`git log`、`git reset`。
- 教学重点：历史记录可以帮助定位变化，但回退操作需要谨慎。
- 通关反馈：玩家学会观察历史并执行恢复类操作。

#### 关卡 6：冲突试炼

- 学习目标：理解冲突产生原因和解决流程。
- 核心命令：`git merge`、编辑冲突文件、`git add`、`git commit`。
- 教学重点：冲突不是错误，而是 Git 需要人类判断保留哪部分内容。
- 通关反馈：玩家第一次完成冲突处理，不再害怕 conflict。

### 7.2 后续章节规划

#### 第一章：入门筑基

- 初始化仓库。
- 查看状态。
- 暂存文件。
- 提交快照。
- 查看历史。

#### 第二章：分支修行

- 创建分支。
- 切换分支。
- 合并分支。
- 删除分支。
- 理解 main 与 feature 分支。

#### 第三章：协作试炼

- clone。
- remote。
- fetch。
- pull。
- push。
- 解决本地与远程进度不一致。

#### 第四章：冲突与恢复

- merge conflict。
- restore。
- reset。
- revert。
- stash。
- reflog。

#### 第五章：高手进阶

- rebase。
- cherry-pick。
- tag。
- bisect。
- 多人协作工作流。

### 7.3 Boss 关设计

每个章节可以有一个综合 Boss 关。Boss 关不引入太多新命令，而是把本章学过的命令组合起来。

示例：分支 Boss 关

1. 从 main 创建 feature 分支。
2. 在 feature 分支提交功能。
3. 切回 main。
4. 合并 feature。
5. 删除 feature 分支。
6. 查看提交历史确认结果。

Boss 关目的：检验玩家是否理解流程，而不是只记住单个命令。

## 8. 成长系统设计

### 8.1 当前规则

当前成长系统可以继续作为 MVP 核心：

- 等级范围：Lv.1 - Lv.10。
- 每级需要 100 XP。
- 通关后根据基础 XP 和评分奖励经验。
- 重复通关默认不重复获得完整 XP。
- 刷新最佳成绩时可以获得额外收益。
- 称号由完成情况和挑战表现解锁。

### 8.2 等级主题

推荐用 Git 技能成长映射等级称号方向：

| 等级 | 阶段名 | 学习阶段 |
| --- | --- | --- |
| Lv.1 | 初入代码山门 | 认识 Git 与仓库 |
| Lv.2 | 暂存入门 | 掌握 add / commit |
| Lv.3 | 分支学徒 | 理解 branch / switch |
| Lv.4 | 合并修士 | 掌握 merge |
| Lv.5 | 历史行者 | 理解 log / reset |
| Lv.6 | 冲突调和者 | 处理 conflict |
| Lv.7 | 协作道友 | 使用 remote / pull / push |
| Lv.8 | 回溯术士 | 掌握 restore / revert / stash |
| Lv.9 | 变基真人 | 学习 rebase / cherry-pick |
| Lv.10 | Git 宗师 | 综合掌握 Git 工作流 |

### 8.3 XP 来源

推荐 XP 来源：

- 首次通关。
- 高分通关。
- 刷新最佳成绩。
- 少用提示。
- 少犯错误。
- 完成章节 Boss 关。
- 完成全章节学习路径。

### 8.4 评分维度

评分应该偏鼓励，不要让新手因为错误而失去继续尝试的动力。

推荐评分维度：

- 错误命令次数。
- 使用提示次数。
- 命令顺序是否合理。
- 是否一次性完成核心步骤。
- 是否使用简洁、符合目标的命令路径。

### 8.5 重复挑战收益

重复挑战的目的不是刷 XP，而是复盘和提升。

推荐规则：

- 已完成关卡可以重复挑战。
- 未刷新最佳成绩时不重复发放完整 XP。
- 刷新最佳成绩时给予差额奖励或少量精进奖励。
- Profile 中展示最佳成绩，鼓励玩家回头优化。

## 9. 称号与成就系统

### 9.1 设计原则

称号系统是这个项目的差异化亮点之一。建议保持中文修仙风格，但称号解锁条件要和 Git 技能绑定。

原则：

- 称号应该表达玩家掌握的能力。
- 称号可以有趣，但不能误导 Git 概念。
- 不只奖励速度，也奖励坚持、复盘和完整学习。
- 称号既是奖励，也是学习路径提示。

### 9.2 称号示例

- 初入仓库：完成第一次 Git 初始化或首次提交。
- 暂存小童：完成暂存区相关挑战。
- 提交术士：掌握基础 commit 流程。
- 分支行者：完成分支创建与切换。
- 合并护法：完成 merge 挑战。
- 冲突调停者：完成冲突解决。
- 回滚道人：完成历史回溯挑战。
- 变基真人：完成 rebase 进阶挑战。
- Git 宗师：完成全部主线挑战。

### 9.3 成就示例

- 第一次提交：完成首个 commit。
- 稳如老狗：连续多个命令无错误。
- 冲突不慌：首次解决 merge conflict。
- 历史观察者：使用 `git log` 完成任务。
- 不破不立：成功使用 reset 或 revert。
- 再战巅峰：重复挑战并刷新最佳成绩。
- 一气呵成：不使用提示完成关卡。

## 10. 新手引导设计

### 10.1 首次进入

首次进入游戏时，可以展示简短说明：

- 这是一个模拟 Git 训练场。
- 玩家需要根据任务输入 Git 命令。
- 输错不会失败，系统会给出解释。
- 完成关卡可以获得 XP 和称号。
- 学习路径会从简单到复杂逐步展开。

### 10.2 第一关引导

第一关应该更像教程：

- 明确解释什么是仓库。
- 明确解释工作区、暂存区和提交。
- 在玩家第一次输入错误时给出温和说明。
- 提示可以更直接，帮助玩家完成第一次成功体验。

### 10.3 渐进式放手

推荐节奏：

- 前 2 关：强引导，提示更具体。
- 第 3-4 关：中等引导，鼓励玩家先尝试。
- 第 5-6 关：减少直接答案，更强调概念理解。
- Boss 关：默认隐藏强提示，但允许主动展开。

## 11. 文案与教学语气

整体语气建议：

- 中文优先。
- 轻松但不幼稚。
- 鼓励式反馈。
- 错误提示教学化。
- 修仙梗用于包装，Git 概念必须准确。

不推荐：

> 命令错误。

推荐：

> 还差一步。当前改动还没有进入暂存区，`commit` 暂时找不到可以提交的内容。先想想如何把文件加入 staged 状态。

不推荐：

> 你失败了。

推荐：

> 这次 Git 没有按预期前进，但你发现了一个常见坑：提交前需要先选择要提交的内容。

## 12. 视觉主题

### 12.1 主题方向

推荐主题：**代码修仙 / 命令江湖**。

这个主题适合当前中文称号系统，也能让 Git 学习过程更有记忆点。

### 12.2 视觉关键词

- 暗色终端。
- Git 橙色或终端绿色强调色。
- 卡片式关卡。
- 命令历史记录。
- 等级进度条。
- 称号徽章。
- 成功反馈光效。
- 柔和错误提示。

### 12.3 颜色建议

- 背景：深蓝黑或暗灰。
- 主色：Git 橙或终端绿。
- 成功：绿色。
- 警告：黄色。
- 错误：柔和红色，避免过度刺眼。
- 稀有称号：紫色或金色。

### 12.4 可读性原则

视觉效果不能影响学习效率：

- 命令文本必须清晰。
- 错误和成功状态必须容易区分。
- 动效不能打断输入。
- 终端区域要保持足够对比度。

## 15. 最重要的设计判断

这个游戏最核心的价值，不是让玩家背下更多 Git 命令，而是让玩家理解 Git 的状态变化。

因此后续所有设计都应该围绕三个问题展开：

1. 玩家现在处于什么 Git 状态？
2. 玩家输入的命令如何改变这个状态？
3. 玩家是否理解为什么下一步要这么做？

只要这三个问题回答清楚，UI、关卡、成长系统和称号系统就会服务于同一个目标：让 Git 从“可怕的黑盒命令”变成“可理解、可练习、可掌握的开发技能”。
